scrcap Windows Port Research

A source-grounded review of the current scrcap-windows/ implementation, comparable Windows screenshot and capture applications, and Windows platform guidance. The goal is to identify simple patterns that preserve performance without turning the port into a large framework.

Executive Summary

The Windows port is already organized in a good way for a small native screenshot tool. The main issue is not architecture. The main issue is that several hot paths convert or retain more data than needed: capture becomes PNG too early, WGC frames pass through several layers before becoming an image, scrolling capture stores full bitmap frames, and pixelate rendering reads pixels cell by cell during render.

The better pattern is a small explicit pipeline: 

capture target -> backend selection -> raw BGRA frame -> editor model -> render/cache -> export PNG only at output boundaries

Mature Windows screenshot and capture projects converge on the same few principles: isolate platform capture code, make backend choice explicit, keep frame data close to its native format, handle DPI and monitor geometry deliberately, put overlay ownership in one place, and test on real Windows graphics paths.

Rank Change Why it matters Expected complexity
P0 Keep current project split and WPF shell. The existing structure already separates portable core logic, Windows platform APIs, and UI composition. Rewriting would add risk without solving the current performance issues. None. This is a constraint on future work.
P1 Replace internal PngBytes capture contract with raw frame/pixel data. Avoids capture -> PNG encode -> PNG decode -> editor render. It also gives WGC and GDI a stable lower-level result shape. Medium. Touches capture service, app handoff, editor load, save/copy/export boundaries.
P1 Make backend choice explicit and observable. Windows capture failures are environment-specific. A user or test should know whether WGC or GDI was used and why fallback happened. Low. Metadata plus a small backend resolver.
P1 Add real Windows fixture tests. Current source-string tests are useful as temporary guardrails but cannot catch actual capture, DPI, overlay, or WPF behavior bugs. Medium. Needs a Windows test fixture and CI/manual gate.
P2 Rework scrolling capture storage. Scrolling capture is the one feature likely to create very tall images and high memory pressure. The cap should govern storage, not only report late failure. Medium to high. Can be done incrementally around current stitching logic.
P2 Cache pixelate output using downscale/upscale. Eliminates repeated CopyPixels, tiny allocations, and brush creation inside render loops. Low to medium. Narrow editor rendering change.
P3 Optimize WGC conversion behind a converter interface. Greenshot and OBS show that direct GPU texture/staging copies are cleaner long-term than layered SoftwareBitmap conversion. Medium. Keep isolated so it does not leak into app logic.

Current Local Baseline

The current scrcap-windows/ subtree has a reasonable shape for a native Windows port.

Keep

  • Scrcap.Core contains portable logic: annotations, settings, keymaps, filename generation, and stitching.
  • Scrcap.Windows.Platform contains Windows capture, monitor/window selection, global hotkeys, tray integration, and taskbar/theme code.
  • Scrcap.Windows.UI contains WPF app composition, overlay, editor, preferences, and resources.
  • Directory.Build.props enables nullable, implicit usings, deterministic builds, and warnings-as-errors.
  • app.manifest declares Per-Monitor V2 DPI awareness.
  • Release scripts already check restore, build, tests, UI color guardrails, and publish artifacts.

Fix

  • CaptureResult carries byte[] PngBytes, so screenshots are encoded as PNG before the editor needs them.
  • WindowsGraphicsCaptureBackend converts Direct3D frames through SoftwareBitmap, WinRT buffers, managed arrays, and System.Drawing.Bitmap.
  • Scrolling capture keeps a growing List<Bitmap> and row hash state until final stitching.
  • Pixelate preview/export performs per-cell CopyPixels and allocates a new 4-byte array in the render loop.
  • Several Windows tests assert source strings rather than exercising WGC, GDI fallback, overlay selection, or packaged app behavior.

External Repository Study

The following projects were studied because they cover the relevant Windows capture design space: screenshot workflow tools, long-running capture/recording, WGC interop, cross-platform overlay capture, and high-performance backend selection. The patterns below are intentionally reduced to what scrcap should copy, not everything these applications do.

ShareX

Source: ShareX at a241c7e5a16b4a40163bf9954a1c06c7b4f3e640.

Relevant files
  • ShareX.ScreenCaptureLib/Screenshot.cs
  • ShareX.ScreenCaptureLib/ScrollingCaptureManager.cs
  • ShareX.ScreenCaptureLib/RegionCaptureTasks.cs
What it does well
  • Region and window capture entry points are small task helpers. They open the picker, wait for a target, and return a concrete result.
  • Classic GDI capture is isolated in a helper around GetWindowDC, compatible DC/bitmap, BitBlt, optional cursor drawing, and cleanup.
  • Scrolling capture compares locked bitmap memory directly instead of using high-level per-pixel APIs.
  • Overlap and stitch logic is optimized around row/bitmap data access instead of UI rendering.
Pattern to copy Keep capture target picking as a concrete workflow. For scrolling capture, prefer locked memory, row hashes, and strip composition over per-pixel object APIs.
Pattern to avoid ShareX has a very large workflow, upload, automation, and destination system. scrcap should not copy that product surface.
scrcap implication Keep GdiCaptureFallback small. Add a CaptureTargetPicker. Rework scrolling capture storage and row access without adding new user-facing settings.

ScreenToGif

What it does well
  • Capture policy is explicit: direct capture, changed-only capture, cached capture, and image capture are separate implementation choices.
  • Frame data can be memory-backed or file-backed. The frame model supports Path and Data.
  • Long-running capture can use buffered and compressed streams, reducing managed object pressure.

Pattern to avoid ScreenToGif is a recorder. Its many recording settings should not become scrcap UI. Use the storage idea only where scroll capture needs it.
scrcap implication ScrollingCaptureOptions.MemoryCapBytes should map to a real storage policy: keep strips in memory under the cap, then spool to temp files with cleanup.

Microsoft Platform Guidance

The platform guidance supports the current overall direction: WGC as the primary API, GDI as a fallback, Per-Monitor V2 DPI awareness, and careful overlay rendering.

Topic Source Practical reading for scrcap
Windows.Graphics.Capture Screen capture with Windows.Graphics.Capture WGC is the right primary API for modern Windows window and monitor capture. It is built for capture items and capture sessions rather than raw desktop scraping.
Threading Direct3D11CaptureFramePool.CreateFreeThreaded CreateFreeThreaded avoids DispatcherQueue dependency and raises frame events from an internal worker thread. The current port uses this right direction.
Session features GraphicsCaptureSession Features such as cursor capture, border requirement, dirty region mode, secondary windows, and minimum update interval are version-dependent. Access them through capability checks.
Cursor capture IsCursorCaptureEnabled Cursor inclusion should be a backend feature gate, not a hard assumption.
Borderless capture IsBorderRequired, GraphicsCaptureAccessKind Suppressing the capture border requires explicit support and access. Use only behind a capability check and expect denial.
DXGI Desktop Duplication Desktop Duplication API, AcquireNextFrame DXGI exposes dirty/move rects, pointer updates, rotation, timeouts, and lower-level GPU surfaces. It is powerful but increases correctness burden. Do not add it until WGC fails a real requirement.
GDI capture GDI image capture BitBlt remains a useful compatibility fallback, especially for older or unusual windows, but should not be the main performance path.
High DPI High DPI desktop app development, Per-monitor DPI-aware WPF guidance Per-Monitor V2 is mandatory for picker/editor correctness because WPF uses device-independent units while capture works in physical pixels.
WPF rendering WPF graphics rendering tiers, Window.AllowsTransparency Transparent topmost WPF overlays are appropriate for selection, but they should stay visually simple and short-lived.
API choice: WGC primary plus GDI fallback is the right simple stack now. DXGI Desktop Duplication should remain a future escape hatch for a proven gap, not a preemptive architecture layer.

What Makes a Good Windows Port

1. Backend strategy is explicit and observable

A good Windows port does not hide backend selection inside broad exception handling. It records the requested backend, chosen backend, capability checks, and fallback reason. This matters because capture failures vary by OS version, GPU driver, window type, desktop state, and policy.

Good shape Auto, WindowsGraphicsCapture, and Gdi are visible concepts. Auto resolves to a chosen backend. The result includes backend metadata.
Bad shape Try WGC, catch broad exceptions, silently fall back to GDI, and return only pixels. This works until users report "capture is slow" or "window capture is wrong" and there is no evidence.
Repo examples OBS logs selected vs chosen method. ScreenToGif selects concrete capture implementations. scrcap should copy this at small scale.

2. Raw frames stay raw until export

PNG is excellent for files and clipboard formats. It is not a good internal message format between capture and editor. The internal result should be BGRA pixels, a WPF BitmapSource, or a wrapper that can produce either without re-encoding.

Good shape Capture returns CapturedPixels or CapturedFrame. Editor receives that object directly. PNG is produced only for save, copy PNG, drag-out, or export.
Bad shape Capture returns PNG bytes. UI decodes PNG immediately. Export later re-encodes PNG after annotation flattening.
Repo examples Greenshot moves from WGC texture to CPU pixels with staging texture copy. OBS keeps textures as textures in the hot path.

3. DPI and monitor geometry are core behavior

Screenshot tools cannot treat DPI as a display styling concern. Selection rectangles, window bounds, captured pixels, virtual screen coordinates, and WPF device-independent pixels must cross one deliberate boundary.

Good shape A tiny helper handles WPF point -> physical pixel, physical pixel -> WPF point, per-monitor scale, and virtual-screen normalization.
Bad shape Coordinates are converted ad hoc inside overlay mouse handlers, capture service, and editor math.
Repo examples Flameshot handles selected-screen geometry and device pixel ratio when placing the capture widget.

4. Overlay ownership is simple and singular

The app composition root should not know every detail of region selection, window picking, countdowns, cancellation, monitor placement, and keyboard interaction. It should ask for a target and then capture it.

Good shape ICaptureTargetPicker.PickAsync(...) returns a CaptureTarget or cancellation. The overlay/controller owns details.
Bad shape App.xaml.cs directly constructs overlay windows, wires each picker path, and accumulates workflow logic.
Repo examples Flameshot uses a capture widget as the owner of capture UX. ShareX uses small region capture task helpers.

5. Long capture has a storage policy

Scrolling capture is not just "take many screenshots and stitch them." It is an unbounded data problem. The port should know when it is storing full frames, strips, hashes, or temp files.

Good shape Keep previous frame, current frame, row hashes, output strips, memory estimate, temp spool state, progress, and stop reason.
Bad shape Retain every full frame until the final output, with the cap checked after large allocations have already occurred.
Repo examples ScreenToGif explicitly supports memory-backed and file-backed frames. ShareX uses locked bitmap memory for scrolling comparisons.

6. Editor effects are cached

Rendering should draw prepared visual assets, not repeatedly sample the source image at a tiny granularity. Pixelate is the clearest current example.

Good shape Pixelate creates a cached bitmap for source version + shape bounds + block size + export scale. Render draws the cached bitmap.
Bad shape Render loops allocate byte arrays and brushes for each cell and call CopyPixels repeatedly.
Repo examples Flameshot pixelates by copy region -> scale down -> scale up.

7. Tests run the Windows behavior

A Windows graphics port should not ship on source assertions alone. Source assertions can catch accidental architecture drift, but they cannot verify WGC, GDI fallback, DPI, transparency, HWND capture, or packaged-app behavior.

Good shape Core tests run anywhere. Windows capture fixture tests run on Windows. UIA smoke tests drive actual app flows. Packaged app smoke tests prove the release artifact opens and handles preferences/editor flows.
Bad shape Tests assert that source files contain strings like StartCapture or OverlayWindow and treat that as release confidence.
Repo examples The mature projects rely on real platform code boundaries. scrcap needs test boundaries that match those platform boundaries.

Implementation Plan

This order keeps changes small and testable. It makes behavior visible first, then removes wasteful conversion, then improves the known hot paths.

P0 Keep the current architecture

Decision Keep C#/.NET 8/WPF. Keep Scrcap.Core, Scrcap.Windows.Platform, and Scrcap.Windows.UI.
Why The performance issues are inside data flow and hot paths, not caused by the project split.
Do not do Do not rewrite to WinUI. Do not add DXGI. Do not add a workflow/plugin system.

P1 Add backend metadata and resolver

Current state WindowsCaptureService tries WGC, catches fallback-eligible exceptions, then uses GDI.
Change Add backend preference, backend used, fallback reason, and capability checks. Return them in capture metadata.
Acceptance A capture result can say "requested Auto, chose WindowsGraphicsCapture" or "requested Auto, WGC failed with access denied, chose Gdi".
Tests Unit-test resolver behavior. Add integration test hooks later to force GDI or WGC when running on Windows.

P1 Replace internal PNG transport

Current state CaptureResult.PngBytes forces encode/decode before editing.
Change Return CapturedPixels from platform code or CapturedFrame at UI boundary. Keep PNG encoding in output paths.
Acceptance Single screenshot capture opens in editor without encoding PNG first. Save/copy still produce identical PNG output.
Tests Existing editor flatten tests should still pass. Add tests for metadata preservation and PNG output boundaries.

P1 Add Windows behavior fixtures

Current state Some tests read source code and assert strings.
Change Add a deterministic fixture window that renders colored regions, text, checkerboards, and known bounds.
Acceptance WGC capture of fixture by HWND verifies expected pixels. GDI fallback verifies expected pixels where supported. Overlay UIA can select a known region and exit cleanly.
Tests Windows-only test project or test category. CI can skip when no interactive desktop is available, but release guardrails should run it on a Windows desktop runner or manual gate.

P2 Rework scrolling capture storage

Current state Scrolling capture stores full frames and accumulated hashes until final stitch.
Change Create ScrollingCaptureSession that owns previous frame, current row hashes, output strips, memory estimate, temp spool, progress, and stop reason.
Acceptance Memory cap is enforced before retaining more data. Huge captures return a structured stop reason or use temp spool. Cancellation cleans up temp data.
Tests Pure tests around row hashing/overlap and memory policy. Windows fixture later for real scrolling target.

P2 Cache pixelate rendering

Current state EditorCanvas.DrawPixelate reads one pixel per cell and allocates during render.
Change Introduce a small PixelateRenderer helper that copies the selected region once, scales down, scales up with nearest-neighbor behavior, and caches by source version and shape properties.
Acceptance Pixelate visual output remains correct. Repeated renders do not call CopyPixels per cell. Export and preview share the same strategy.
Tests Rendering test for deterministic pixelate output. Optional benchmark for large pixelate rectangle.

P2 Extract capture target picker

Current state App.xaml.cs directly creates overlay windows for region and window picking.
Change Add CaptureTargetPicker in the overlay/UI boundary. It owns region selection, window picking, selected monitor, countdown, cancellation, and DPI conversions.
Acceptance App.xaml.cs asks for a target and then invokes capture. Overlay behavior remains unchanged.
Tests Unit tests for target picker state decisions where possible. UIA smoke for the actual overlay.

P3 Optimize WGC conversion behind a stable interface

Current state WGC frames convert through SoftwareBitmap and multiple managed copies.
Change Add IFrameConverter. Keep current converter initially if needed. Add staging texture converter once raw frame contract and fixture tests exist.
Acceptance Converter can be benchmarked and replaced without changing capture service, editor, or app composition.
Tests Fixture capture pixel tests compare converter output. Benchmark tracks single-region and full-screen latency.

Suggested Contracts

These are intentionally small. The goal is to make the important boundaries explicit without building a large abstraction layer.

Backend metadata

public enum CaptureBackendPreference
{
    Auto,
    WindowsGraphicsCapture,
    Gdi
}

public enum CaptureBackendUsed
{
    WindowsGraphicsCapture,
    Gdi
}

public sealed record CaptureMetadata(
    CaptureBackendPreference RequestedBackend,
    CaptureBackendUsed BackendUsed,
    string? FallbackReason,
    RectInt32 CaptureBounds,
    double DpiScaleX,
    double DpiScaleY,
    DateTimeOffset CapturedAt);

Raw frame transport

public sealed record CapturedPixels(
    ReadOnlyMemory<byte> Bgra32,
    int PixelWidth,
    int PixelHeight,
    int Stride,
    CaptureMetadata Metadata);

public sealed record CaptureResult(
    CapturedPixels Pixels,
    CaptureTarget Target,
    CaptureMetadata Metadata);

If WPF integration is preferred at the UI boundary, keep that conversion outside the platform service: 

public sealed record CapturedFrame(
    BitmapSource Bitmap,
    CaptureTarget Target,
    CaptureMetadata Metadata);

Capture target picker

public interface ICaptureTargetPicker
{
    Task<CaptureTarget?> PickAsync(
        AppAction action,
        Settings settings,
        CancellationToken cancellationToken);
}

The picker should own overlay windows and return a target. It should not capture pixels. This keeps UI selection and platform capture separate.

Scrolling capture state

public enum ScrollingCaptureStopReason
{
    Completed,
    BottomReached,
    Cancelled,
    Timeout,
    MaxHeightReached,
    MemoryCapReached,
    AlignmentFailed,
    CaptureFailed
}

public sealed record ScrollingCaptureProgress(
    int FrameCount,
    int OutputHeight,
    long EstimatedBytes,
    bool IsSpoolingToDisk,
    ScrollingCaptureStopReason? StopReason);

Pixelate cache key

public readonly record struct PixelateCacheKey(
    long SourceVersion,
    Int32Rect SourceBounds,
    int BlockSize,
    double ExportScale);

The cache can be very small. It only needs to avoid recomputing the same shape while the user pans, redraws, exports, or copies.

Verification and Tests

The Windows port needs a layered verification plan. The current source checks can remain temporarily, but release confidence should come from runtime behavior.

Current host commands

Use these for the existing macOS/native scrcap code when the change touches shared behavior or the root app:

swift run scrcap-core-tests
scripts/test.sh

Windows port commands

Run these from scrcap-windows/ on Windows:

dotnet restore .\Scrcap.sln
dotnet test .\tests\Scrcap.Core.Tests\Scrcap.Core.Tests.csproj
dotnet test .\tests\Scrcap.Windows.Platform.Tests\Scrcap.Windows.Platform.Tests.csproj
dotnet build .\src\Scrcap.Windows.UI\Scrcap.Windows.UI.csproj
pwsh .\tools\Check-HardcodedUiColors.ps1
pwsh .\tools\Test-ReleaseGuardrails.ps1 -SkipPublish

Suggested benchmark budgets

These are starting budgets, not final product claims. Tune them after measuring on actual Windows hardware.

Operation Initial budget Notes
Small region WGC capture < 150 ms after target selection Measure capture service only, excluding human selection time.
Full 4K monitor WGC capture < 300 ms Current SoftwareBitmap path may exceed this. Use as a reason to benchmark before optimizing.
Editor open after capture < 150 ms for common screenshot sizes Removing PNG encode/decode should help here.
Pixelate redraw for large region < 16 to 33 ms after cache warm Cache misses can be slower; repeated renders should be cheap.
Scrolling capture memory Never exceed configured cap by more than one frame/strip Track estimated bytes and spill/stop before retaining too much data.

What Not To Do Yet

  • Do not add DXGI Desktop Duplication until WGC plus GDI fallback fails a real requirement.
  • Do not replace WPF with WinUI or another UI stack. The current bottlenecks are data flow and rendering details, not the UI framework choice.
  • Do not add ShareX-style workflow/upload automation.
  • Do not expose ScreenToGif-style performance settings. Keep storage policy internal unless users need a knob.
  • Do not copy OBS's render graph, game capture, or audio/video architecture.
  • Do not make source-string tests the final release gate.
  • Do not optimize WGC conversion before adding metadata and test fixtures. Otherwise there is no clean way to prove behavior stayed correct.
