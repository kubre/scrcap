01

Vision & Principles

Everything in this plan derives from four non-negotiable principles. When a decision is ambiguous, these break the tie.

P1Speed is the feature

Hotkey-to-crosshair in under 50 ms, capture-to-editor in under 150 ms. Anything that threatens this (webview runtimes, heavyweight frameworks, lazy-loaded toolbars) is rejected, including for cross-platform reasons.

P2Tiny footprint

Shottr is 2.7 MB and proves it's possible. Target: <5 MB binary (stretch: 3 MB), <30 MB idle RAM, 0% idle CPU. The app is a menu-bar resident — it should be invisible until summoned.

P3Opinionated defaults, total remappability

The workflow described in this spec — draw-stays-armed, Esc copies & closes, red by default — ships as the default behavior, not buried config. But every shortcut and behavior is remappable for everyone else.

P4macOS first, portable core

v1 is pure native macOS. But the annotation model, document format, settings schema, and keymap engine are written UI-agnostic so a future Windows/Linux port reuses the brain and only rewrites the hands. Separate native frontends per OS are acceptable if that's what best preserves P1 and P2.

Scope guardrails (v1)

In: region / window / fullscreen / scrolling capture, annotation editor (arrow, rectangle, counter), 7-color palette, full shortcut remapping, clipboard & PNG export, preferences window.

Out (deliberately, for now): OCR, pin-to-screen, color picker, blur/pixelate, ruler, cloud upload, video recording. These are Shottr features worth revisiting in v2 — listed in the roadmap as candidates — but v1 ships the 90% workflow perfectly rather than the 100% workflow adequately.

Capture Modes

Four modes, each bound to its own global hotkey. Defaults avoid collisions with macOS's built-in ⌘⇧3/⌘⇧4/⌘⇧5; all remappable to any combination (see §07).

Default behavior

Every capture opens the editor. A per-mode toggle in Preferences switches any mode to copy-only (straight to clipboard, no editor) — the Shottr behavior you rely on, made configurable per mode rather than globally.

Permissions flow

  • Screen Recording (TCC) — required for any capture on modern macOS. First launch shows a one-screen onboarding that deep-links to the System Settings pane and live-detects when permission lands (no "restart the app" dance).
  • Accessibility — required only for scrolling capture (synthetic scroll events). Requested lazily, the first time the user invokes scrolling capture, never up front.

The one big behavioral decision

Rule — draw stays armed

After you draw an arrow, rectangle, or counter, the shape is committed immediately and the tool stays armed. There is no selection state, no move/resize handles, no Esc-to-deselect step. Got it wrong? ⌘Z and draw again. This inverts Shottr's default — and because there is no "selected object" state, Esc is freed up to mean exactly one thing: done.

This isn't just preference — it simplifies the architecture. The annotation layer becomes an append-only stack of vector shapes with an undo pointer. No hit-testing, no transform handles, no selection rendering. Less code, fewer bugs, smaller binary. (A move/edit mode can be added later as an opt-in without disturbing this model: shapes stay vector until export.)

Rendering model

  • Bitmap layer: the screenshot, drawn once into a CALayer. Never re-encoded until export.
  • Vector layer: annotations as a [Shape] array rendered via Core Graphics on top. Each shape: {kind, color, geometry, seq} — pure data, no AppKit types. This is the portable core (P4).
  • Undo: a cursor into the shape array. Undo = decrement, redo = increment, new draw truncates redo tail. Trivially correct.
  • Export: flatten bitmap + vectors at native scale (Retina-aware: capture at 2×, annotate in points, export at 2×).

The v1 stack, concretely

  • Language: Swift 5.10+, AppKit. No third-party dependencies at all in v1 — every dependency is binary size and supply-chain surface. (SwiftUI allowed only for the Preferences window, where latency is irrelevant; it adds ~0 size since the frameworks ship with macOS.)
  • Capture: ScreenCaptureKit (SCScreenshotManager on macOS 14+) with a CGWindowListCreateImage fallback for macOS 12–13. Min target: macOS 12.3.
  • Hotkeys: Carbon RegisterEventHotKey — ancient but still the only API that delivers reliable system-wide hotkeys without requiring Accessibility permission or an event tap.
  • App shape: LSUIElement menu-bar app — no Dock icon, no main window. Optional launch-at-login via SMAppService.
  • Distribution: direct download, Developer ID–signed + notarized DMG. Not the App Store for v1 — sandboxing breaks scrolling capture's synthetic scroll events and complicates TCC flows.
Portability discipline (the P4 tax we pay now)

Three modules are written as pure Swift with zero AppKit imports, with the spec documented in docs/core-model.md: the annotation model (shapes, undo), the keymap engine (action ↔ chord mapping, conflict detection), and the settings schema (versioned JSON). These are the parts a Rust or per-OS rewrite would otherwise re-invent badly. Cost today: near zero. Value later: the port starts at 40% done.

06

Scrolling Capture Design

The hardest feature in the app, so it gets its own design and its own milestone — after the core workflow already ships.

Algorithm: scroll-and-stitch

  1. Select — user picks a region or window (reusing the M1 overlay).
  2. Capture frame 0, then inject a smooth synthetic scroll (CGEvent scroll-wheel events targeted at the region's center) of ~70% of the visible height.
  3. Settle & capture — wait for two consecutive identical frames (handles bounce, lazy-loading, animations) with a 600 ms timeout, then capture frame N.
  4. Match overlap — compute per-row hashes of the new frame, find the best alignment against the previous frame's tail via normalized cross-correlation on the hash sequence. Sub-region matching ignores fixed headers/footers (detected as rows identical across all frames — they get cropped from middle frames automatically).
  5. Terminate when two consecutive frames produce zero new rows (bottom reached), the user presses Esc/clicks Stop, or a configurable max-height (default 20,000 px) is hit.
  6. Composite into a single bitmap and hand to the standard editor — annotations work identically on tall images; the editor gains vertical scroll.

Known hard cases & mitigations

CaseMitigation
Sticky headers / floating elementsCross-frame identical-row detection; crop from all but first frame
Lazy-loading content (infinite feeds)Settle detection + max-height cap; live progress preview so the user can stop anytime
Elastic bounce at document endBounce produces negative-offset match → treated as terminal frame
Non-scrollable target pickedFrame 1 == frame 0 → bail gracefully with a toast, deliver a normal capture instead
Memory on very tall pagesStitch incrementally into one growing buffer; 20k px × 2x-Retina ≈ 450 MB worst case → cap + warn
07

Settings & Shortcut Remapping

Every action in the app — global and in-editor — routes through the KeymapEngine, so remapping is universal by construction, not feature-by-feature.

Preferences window (4 tabs)

  • General — launch at login, after-capture behavior per mode (open editor / copy-only / both), save folder & filename pattern, image format.
  • Shortcuts — a recorder control per action (click, press any combination, done). Live conflict detection against both scrcap's own map and known macOS system shortcuts, shown inline in red. Built in-house (~200 lines) rather than importing a dependency.
  • Editor — palette editor (reorder/replace the 7 colors; slot 1 stays the boot default), stroke width, counter style, Esc behavior (copy+close / close only), arrow style.
  • Advanced — Retina export scale, scrolling-capture max height, reset to defaults.

Persistence

One versioned JSON file: ~/Library/Application Support/scrcap/settings.json with a schemaVersion field and explicit migration functions. Human-readable and hand-editable on purpose — power users (you) can sync it in dotfiles. Writes are atomic (temp file + rename).

Performance & Size Budgets

Budgets are CI-enforced from M0 — a failing budget blocks merge, the same as a failing test. This is how the app stays at Shottr weight instead of drifting.

Hotkey → selection overlay visible< 50 ms16 msOverlay window pre-created at launch and kept ordered-out, not built on demand

Two design choices do most of the work here: pre-warmed windows (the overlay and an empty editor exist at launch, hidden — showing a window is fast, creating one is not), and zero dependencies (nothing to lazy-load, nothing to bloat).

Cross-Platform Strategy (later)

Explicitly deferred until macOS v1 is shipped and stable — per your priority call. But the door is designed now so it doesn't have to be forced later.

What each OS needs replaced

LayerWindowsLinux
Capture APIWindows.Graphics.Capture / DXGI duplicationXDG Desktop Portal (Wayland) · XShm (X11)
Global hotkeysRegisterHotKeyportal GlobalShortcuts (Wayland) · XGrabKey (X11)
Overlay + editor UIWin32/WinUI or Rust-native (e.g. wgpu)layer-shell surface · GTK or Rust-native
ClipboardWin32 clipboardwl-clipboard / X selections
Annotation model, keymap, settings— ported once from the documented portable core, shared thereafter —

Decision framework (to be made then, not now)

  • Path A — separate native apps per OS sharing the documented core model. Best feel, best size on every OS; highest total effort. You already said this is acceptable if it preserves performance.
  • Path B — Rust core + thin native shells. Rewrite the portable core once in Rust, keep UI native per OS. Middle effort, keeps P1/P2 intact. The likely winner if Windows + Linux are both wanted.
  • Path C — single cross-platform UI toolkit. Only if A and B prove impractical, and only with a toolkit that demonstrably meets the §08 budgets. Electron remains permanently out.

The trigger to revisit: macOS v1 at M5, stable for a month of daily use, and actual demand for another OS.
